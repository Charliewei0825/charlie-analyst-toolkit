# Charlie · 分析师工具包 SOP（六模式）

> **本 skill 提供六种输出，启动时先确认用户要哪一种。**

## 模式选择

| 模式 | 你手头有什么 | 你需要什么 | 输出物 |
|---|---|---|---|
| **A · 访谈提纲** | 知识库目录（研报/纪要/模版 PDF）+ 访谈对象 + 覆盖标的 | 跟这个人开会时要问的问题清单 | Markdown 提纲 + PDF，三组件结构（背景→双语提问→缺口表） |
| **B · 全覆盖观点汇总** | 某分析师对多家公司的长面板研报（每家 6-16 份不等） | 理解该分析师的全部覆盖观点——哪些 top pick、哪些观点变了、哪些超预期/低于预期、近期催化剂日历 | Markdown 全景文档 + PDF |
| **C · 研报 PPTX** | 目标公司研报 PDF 文件夹 + 参考 PPTX 模版 | 一份结构化公司深度研报 slides，格式严格匹配参考模版 | 32-41 页 PPTX + Markdown 文字底稿 + 图片建议清单 |

### 怎么判断用哪个

- 用户说 "帮我准备访谈问题"、"我要跟 XX 开会问什么"、"手持问题清单" → **模式 A**
- 用户说 "帮我总结一下这个分析师的整体观点"、"他覆盖了哪些公司"、"他在哪些标的上变了观点"、"过去一年的评级变化"、"哪些是 top pick"、"他看错了哪些" → **模式 B**
- 用户说 "写研报"、"做一份公司 PPT"、"做 slides"、"公司深度"、"分析一下 XX 公司做份 deck" → **模式 C**
- 用户同时有访谈需求 + 也想了解全覆盖背景 → **先 B 后 A**（先读完全部研报→输出全景文档→再基于全景文档生成访谈提纲）

## 模式 A · 访谈提纲生成

> **触发场景**：需要基于知识库生成访谈手持问题清单。访谈对象可以是卖方分析师、公司管理层（CEO/CFO/CMO/IR）、KOL 医生、机构股东等。
> **输入**：知识库目录（研报/纪要/模版 PDF）+ 访谈对象 + 覆盖标的或管线
> **输出**：Markdown 提纲 + PDF，严格对标模版的三组件结构（背景与预期 → 双语提问 → 查缺补漏扫描表）

## 一、预处理 · 知识库分类与事实基线

### 1.1 知识库三层分类法

拿到一个课题目录后，第一件事是把所有文件按来源分成三层。**不同访谈类型，L1/L2 的身份会变：**

| 访谈对象 | L1（此人的观点） | L2（别人的观点） | L3（事实背景） |
|---|---|---|---|
| 卖方分析师 | 该分析师的研报 PDF | 其他分析师/公司管理层/股东的会议纪要 | 模版 PDF、NCT、行业报告 |
| 公司管理层 | 该公司过往的业绩会 transcript、投资者日 presentation、股东信 | 其他公司/分析师/竞争者的评论 | 模版 PDF、NCT、行业报告 |
| KOL/医生 | 该医生过往的访谈记录、发表的文章 | 其他医生/分析师的观点 | 模版 PDF、临床指南、文献 |
| 机构股东 | 该基金的 13F、股东信、过往访谈记录 | 其他股东/分析师的观点 | 模版 PDF、持仓数据 |

**红线规则：** L2 里的话永远不能安到 L1 访谈对象头上。比如分析师访谈中，"Pete 在 Basma 纪要中说"——Basma 是 Leerink 的分析师，不是 Pete。公司访谈中，"CEO 在 Andrew Tsai 纪要中说"——那是 Jefferies 分析师的转述，不是 CEO 原话。

### 1.2 逐份材料阅读 SOP（不能跳，不能委托）

如果访谈对象是**分析师**，L1 = 其研报 PDF；如果是**公司管理层**，L1 = 其过往 transcript/presentation/shareholder letter。

```
for 每个 L1 文件:
  if pdf:  pdftotext "$f" /tmp/out.txt
  if docx: textutil -convert txt "$f" -output /tmp/out.txt
  Read /tmp/out.txt              # 逐份读，记录到追踪表
```

**每份报告必须提取的字段（录入追踪表）：**

| 字段 | 示例 |
|---|---|
| 日期 | Jun 22, 2026 |
| 标题 | "LSDelivering the Goods: DT120 Trips Passed Expectations in P3 EMERGE" |
| 作者 | Pete Stavropoulos, Ph.D. |
| 评级/PT | Overweight / No PT |
| 报告时股价 | $24.48 |
| 核心 thesis（1 句话） | EMERGE 数据远超预期，MDD PoS 上调至 80% |
| 模型参数变化 | DCF NPV $6.3B → $7.8B；peak sales >$2.3B；折现率 12% 不变 |
| 最新数据点 | 5.8hrs 达离院标准；53% 入组为 TRD；1-2 doses in first 6 months |

**全部读完后，输出该分析师的 PT/DCF/关键参数变化时间线**（见下文 1.3）。

### 1.3 参数时间线追踪

读完所有报告后，整理一份时间线，一眼能看出观点的演变：

```
[分析师名] 对 [标的] 的 DCF NPV 演变：
  Jul 31, 2025: $1.8B（18% 折现率，仅 GAD）
  Feb 26, 2026: $3.6B（折现率 18%→12%，Voyage power >99%）
  Apr 22, 2026: $6.1B（渗透率上调 + PTSD 新增）
  Jun 22, 2026: $7.8B（EMERGE 读出，MDD PoS 60%→80%）

[分析师名] 对 [标的] 的 PT 演变：
  Jun 23, 2025: $8（COMP005 后下调）
  Nov 4, 2025:  $20（时间线加速 9-12 月）
  May 13, 2026: $25（折现率 12%→11%，launch 提前至 2027）
```

**以最新报告的参数为准。** 如果 8 个月前报告说 PT $20、最新报告说 $25，提纲里必须用 $25。

### 1.4 联网事实基线（必做）

以下数据点必须联网核实，不能凭记忆或旧报告：

```
必须搜索的维度（使用 Tavily/WebSearch）：
  - 各标的当前股价、市值（标注查询日期）
  - 最新融资动态（增发/ATM/现金余额）
  - 关键监管进展（FDA 会议是否已开、clinical hold 是否已解）
  - 其他投行的最新评级和 PT（作为参照系）
  - 关键催化剂是否已发生（"Q3 读出"——现在是 Q3 了吗？读了吗？）
```

### 2.0b 层级规则速查

| 元素 | MD 语法 | 用途 | 渲染后在 PDF 中的表现 |
|---|---|---|---|
| 文档标题 | `#` | 全文唯一一个 | 深蓝色、底部 2.5pt 实线分隔 |
| 元数据区 | `>` blockquote 连续多段 | 分析师画像、核心论点、时点 | 浅蓝灰底、左侧 4pt 深蓝竖线 |
| 主分区 | `##` | Core / Secondary / Pocket / 第三部分 | 深蓝底白字通栏 |
| 单个问题 | `### **QX** (PX) — 标题` | 每个问题一个 | 浅蓝底、左侧 3pt 深蓝竖线 |
| 背景段落 | `>` blockquote | 放在 `###` 和 `**"` 之间 | 浅蓝灰底 |
| 英文提问 | `**"..."**` | 紧接在 blockquote 后面 | 粗体 |
| 中文翻译 | 普通文本 | 英文提问下一行 | 正常字重 |
| 缺口标签 | 普通文本 | `关联缺口：G-X 🚨` | 正常字重 |
| 表格 | `| ... |` | Gap 矩阵和补充检查项 | 深蓝表头、7pt 字号 |
| 脚注 | `*...*` | 全文最后 | 斜体 |

### 2.0c `$` 符号处理规则

美元金额在 MD 中会被某些渲染器（pandoc、Typora）误解析为 LaTeX 数学公式。处理方式：

```
在 MD 源文件中（Typora 编辑时）：
  对裸 $ 金额使用反斜杠转义：\$1.69B、PT \$25、~\$30M
  
  不转义的情况：
  - $50B TAM（$ 前面没有数字/字符，不会被误判为公式）
  - 已经用反斜杠转义的
  
在 PDF 生成命令中：
  pandoc 加 -f markdown-tex_math_dollars 作为双保险
```

**为什么这样做**：Typora 预览时 `$1.69B` 可能渲染异常（闪一下公式模式），用 `\$1.69B` 则稳定显示。pandoc `-f markdown-tex_math_dollars` 保证即使有个别漏网之鱼也不会在 PDF 中乱码。

## 三、提纲结构与内容设计原则

### 2.1 三组件结构（对标 ziltivekimab 模版）

每个正式问题**必须**包含以下三段：

```
【组件一 · 背景与预期】
> **背景与预期**:
> [已知数据] —— 来自研报和联网的事实基线
> [该分析师的观点] —— 来自 L1 研报的直接引用
> [预期他可能怎么回答] —— 给 PM 一个参照锚，便于访谈中判断对方是否偏离了自己的研报逻辑
> [为什么这题重要] —— 关联到哪个信息缺口
> [来源标注] —— 【来源：报告名，日期】

【组件二 · 双语提问】
**"English question in bold?"**
中文紧随其后。
关联缺口：G-X 🚨

【组件三 · 缺口映射】
查缺补漏扫描表见第三部分
```

### 2.2 问题分层设计

```
Core（必问，P1）  —— 当前时点最紧迫、只能问这个分析师的问题（3-5 个）
Secondary（P2）   —— 标的覆盖相关，重要但可压缩（3-5 个）
Pocket Questions  —— 暖场/收尾/探测真实情绪（2-3 个，选当天氛围合适的）
```

每道题标 `(P1)` 或 `(P2)`，PM 可以根据时间取舍。

### 2.3 八维度覆盖检查

写完后逐维打勾——每个维度必须至少有 1 个正式问题。这八个维度按访谈对象调整权重：

| # | 维度 | 问分析师 | 问公司管理层 |
|---|---|---|---|
| ① | 临床数据读出 | 你的模型里 PoS 怎么设？time to readout 的假设？ | 入组进度？数据锁了吗？准备在哪家大会读？ |
| ② | 疗效假设验证 | 你的 base case 效应量是多少？和市场共识的差距？ | Ph2 的数据你们内部怎么看？有没有看到亚组信号？ |
| ③ | 安全性底线 | 你最盯哪个安全信号？什么水平你会改 rating？ | 盲态下有没有看到令人担心的 AE 不平衡？ |
| ④ | 竞争护城河 | 这几个分子/标的你的排序？如果只留一个 OW 你留谁？ | 你们和竞品的差异化在哪？对方的数据你们怎么看？ |
| ⑤ | 商业化路径 | peak sales 的渗透率假设怎么拆？定价锚点？ | launch 准备到哪了？销售团队多少人了？payer 谈了吗？ |
| ⑥ | 监管与申报 | NDA timeline 的 base case？CRL 风险在哪？ | FDA 最近一次沟通说了什么？label 你们预期什么样？ |
| ⑦ | 管线协同价值 | 第二/第三适应症在你的 DCF 里占多少？ | 下一个适应症的 trial 什么时候开？ |
| ⑧ | 公司战略优先级 | 你觉得他们该自己卖还是被收购？ | BD 合作谈了吗？M&A 兴趣有吗？现金跑道到什么时候？ |

**分析师访谈侧重** ①②④⑧（外部视角、模型参数、排序），**公司访谈侧重** ①③⑤⑥⑦（内部信息、执行细节）。根据对象调整每维的问题数量。

### 2.4 问题设计原则

```
✅ 好的问题：
  - 基于该分析师自己研报里的具体数字追问（"你在 2 月报告里给 DCF $3.9B，现在 EMERGE 后是否翻倍？"）
  - 让他做选择题/排序（"如果只留一个 OW，你留哪个？"）
  - 探测他报告中回避的矛盾（"你说 functional unblinding 不影响结论，但 EMERGE 无低剂量组——这个论证还成立吗？"）

❌ 差的问题：
  - "你怎么看这家公司？"（太宽泛，对方会给标准话术）
  - "能介绍一下你们的覆盖吗？"（浪费宝贵的私人访谈时间）
  - 和该分析师的覆盖标的无关（问 Pete CMPS 的问题，那是 Josh 覆盖的）
```

## 四、引用与来源标注规范

| 症状 | 诊断 | 修复 |

## 七、示例：典型错误与正确做法（以某次 Cantor 致幻剂任务为例）

### 例 1：用旧报告的模型数字

| ❌ 错误 | ✅ 正确 |
|---|---|
| "Pete 的 DCF NPV ~$3.9B"（来自 Feb 1 报告） | "Pete 的 DCF NPV $7.8B"（来自 Jun 22 最新报告，EMERGE 后上调） |

**做法**：读完所有报告后先画时间线。如果一个参数在 5 份报告中变了 3 次，用最后一次的值。

### 例 2：观点归属张冠李戴

| ❌ 错误 | ✅ 正确 |
|---|---|
| "Pete 在 Basma 纪要中认为 functional unblinding 会放大效应" | "Leerink 分析师 Basma（5/21 纪要）提到如果有低剂量组就不会太担心 functional unblinding——注意这是 Leerink 的观点，可以去问 Pete 是否同意" |

**做法**：纪要 Pool（L2）中所有引用，先确认发言人是谁，再决定怎么用。如果发言人是别的投行的分析师，可以作为"别人怎么看你的标的"来问。

### 例 3：市值凭记忆写

| ❌ 错误 | ✅ 正确 |
|---|---|
| "CMPS 市值仅 ~$800M" | "截至 June 24, CMPS 市值 $1.69B"（联网查询 companiesmarketcap.com） |

**做法**：所有当前市值/股价 → 必须 `tavily_search` 获取当日或最近交易日数据，标注查询日期。几个月前的报告里写的市值是当时的值，不是现在的值。

### 例 4：只读了一份最有名的报告

| ❌ 错误 | ✅ 正确 |
|---|---|
| 读了 "Smoking the Toad"（Jun 2025），然后用它的 DCF $25 PT 写了全部 GHRS 问题 | 追踪到 May 14, 2026 最新报告，PT 已上调至 $35，DCF $2.6B→$3.6B |

**做法**：按文件日期排序，最新那份才是分析师当前的真实观点。12 个月前的老报告里的数字大概率已经过时。

## 模式 B · 分析师全覆盖观点汇总

> **触发场景**：用户拿到了某分析师的研报批量文件（覆盖多个标的、跨越 1-3 年），需要一份全景文档，把该分析师对所有标的的观点演变、确信度排名、催化剂日历、beat/miss 记录一次性理清。
> **典型用户问题**："他目前对哪些公司是强烈推荐？""哪些公司的观点在过去 1 年发生过比较大的变化？""哪些持续在超预期？哪些持续在低于预期？"
> **输入**：该分析师的研报目录（按标的分子文件夹，每个文件夹内含 6-16 份不等的历史报告）
> **输出**：一篇 Markdown 全景文档 + PDF，包含 7 个固定板块

### B.0 批量 PDF 提取（同模式 A 零节）

### B.1 逐标的研报阅读 SOP

对每个标的子文件夹，按文件日期排序，**逐份**读取 txt 并提取以下字段（录入追踪表）：

```
  - 现金/融资历史表
  - 收入模型（适应症 × 峰值销售 × PoS × 风险调整）
  - 保守/乐观情景差异分析

板块 7：估值分析（3 页）
  - 券商估值方法论对比表
  - 敏感性分析表（PoS × WACC → 每股价值）
  - 估值总结（Bull/Base/Bear 三种情景 + 核心二元事件）
```

### C.3 内容密度与深度标准

**纯正文页（无大表格）**：
- 2-4 段正文，600-1300 字
- 每段 4-8 句，包含具体数字、试验名称、文献出处
- 不是项目符号的堆砌，而是完整的分析性段落

**表格页**：
- 表格行数 4-9 行，列数 2-6 列
- 单元格内支持多行文本（`\n` 换行）
- 列宽按内容比例分配

**数据要求**：
- 每个关键声明必须有具体数字（百分比、金额、人数、日期）
- 临床试验数据标注 n 值、治疗期、主要终点和 p 值
- 估值数据标注哪家券商、哪个日期的报告

### C.4 文风规则（模式 C 专属，必须全过程执行）

```
☐ 直接、朴实、具体——只陈述事实和数字
☐ 禁止使用比喻（metaphor）、明喻（simile）、暗喻、排比（parallelism）、夸张（hyperbole）
☐ 禁止诗意化、抽象化、装饰性语言
☐ 减少箭头（→）和破折号（——）的使用
   - 信号传导路径中保留箭头（如 NLRP3→IL-1β→IL-6）
   - 其余语境用逗号、句号或文字连接
☐ 具体禁止句式：
   - "打开了...的大门" ❌ → 直接陈述结果 ✅
   - "堪称惊人/令人震惊" ❌ → 直接写数字 ✅
   - "像...一样"（明喻） ❌ → 直接描述功能 ✅
   - "为...奠定了坚实的基础" ❌ → 直接写"支持"、"验证" ✅
   - "在...的舞台上" ❌ → 直接说"在XX赛道中"或"在XX行业内" ✅
   - "里程碑式的" ❌ → 直接说明事件和影响 ✅
☐ 每个数据点标注来源报告名称和日期
☐ 英文术语保留原文，首字母大写（如 NLRP3、hsCRP、Ph2a、MACE）
☐ 数字格式约定：
   - 金额：$1.2B / $385M / $19.84（股价）
   - 百分比：86%（不是 86 个百分点）
   - 日期：2026 年 Q3 / Mid-2026 / 2026 YE
   - 统计量：HR 0.85 (p=0.021) / n=14
```

### C.6 PPTX 生成技术要求

**工具选择**：
- 使用 `python-pptx`（Python），不使用 `pptxgenjs`（Node.js）——JS 处理中文有编码坑，Python 原生支持 UTF-8。
- 安装：`pip install python-pptx`

**构建方式**：
- 使用空白 layout `prs.slide_layouts[6]` 逐页构建
- 所有字体、字号、颜色通过 `add_textbox` 显式设置，不依赖母版继承
- 表格用 `add_table` 创建，表头和数据行分别设置相同的字体参数
- 文件名格式：`[Ticker]_[公司名]_Research.pptx`

**代码示例（核心模式）**：

```python
from pptx import Presentation
from pptx.util import Inches, Pt
from pptx.dml.color import RGBColor
from pptx.enum.text import PP_ALIGN

prs = Presentation()
prs.slide_width = Inches(13.3)
prs.slide_height = Inches(7.5)

# 添加空白页
slide = prs.slides.add_slide(prs.slide_layouts[6])

# 章节标签
txBox = slide.shapes.add_textbox(Inches(0.56), Inches(0.36), Inches(10.66), Inches(0.65))
tf = txBox.text_frame; tf.word_wrap = True
p = tf.paragraphs[0]; p.text = "行业"
p.font.name = '华文楷体'; p.font.size = Pt(24); p.font.bold = False

# 页面标题
txBox = slide.shapes.add_textbox(Inches(0.56), Inches(1.02), Inches(12.25), Inches(0.65))
tf = txBox.text_frame; tf.word_wrap = True
p = tf.paragraphs[0]; p.text = "这里是页面标题"
p.font.name = '华文楷体'; p.font.size = Pt(20); p.font.bold = True

# 正文（JUSTIFY 对齐）
txBox = slide.shapes.add_textbox(Inches(0.56), Inches(1.60), Inches(12.25), Inches(4.5))
tf = txBox.text_frame; tf.word_wrap = True
for i, text in enumerate(body_paragraphs):
    if i == 0: p = tf.paragraphs[0]
    else: p = tf.add_paragraph()
    p.text = text
    p.font.name = '华文楷体'; p.font.size = Pt(18); p.font.bold = False
    p.alignment = PP_ALIGN.JUSTIFY
    p.space_after = Pt(6)

# 表格（无填充）
tbl_shape = slide.shapes.add_table(nrows, ncols, Inches(left), Inches(top), Inches(w), Inches(h))
tbl = tbl_shape.table
for ri in range(nrows):
    for ci in range(ncols):
        cell = tbl.cell(ri, ci)
        p = cell.text_frame.paragraphs[0]
        p.text = cell_text
        p.font.name = '华文楷体'; p.font.size = Pt(18); p.font.bold = False

prs.save('output.pptx')
```

## 模式 D · 单篇深度转述

> **触发场景**：用户有一篇英文卖方研报 PDF（或行业报告、临床论文），需要转成结构化中文摘要给领导/团队阅读。领导不想读英文，但需掌握全部关键信息——不是简单翻译，而是重建信息架构、补充背景、核实事实。
> **典型用户问题**："领导不想读英文，把这份报告给他重新讲一遍""翻译这篇报告，加点背景""中文总结一下这篇""帮我读一下这份研报，提炼要点"
> **输入**：单篇英文研报 PDF
> **输出**：结构化中文 Markdown 文档（放桌面）+ 可选 PDF

### D.0 与其他模式的关系

模式 D 是四个模式中**最轻量**的——单篇输入、单次输出、不需要知识库目录结构。它共享通用基础设施（PDF 提取、联网事实基线、文风规则、来源标注）。

与模式 B 的关键区别：
- B：一个分析师对多个标的的多份报告 → 全景观点演变
- D：一篇报告 → 结构化中文深度转述

### D.1 工作流程（4 步）

**Step 1 — 提取 PDF 全文**

```bash
pdftotext "[报告路径]" /tmp/report.txt
```

检查行数和文件大小确保提取完整。扫描版 PDF 需 OCR。

**Step 2 — 联网核实事实基线**

对报告中的关键事实做硬核实：
- 当前股价、市值（报告里的数字可能已过时）
- 催化剂状态（"Q3 读出"——现在是 Q3 了吗？读了吗？结果是什么？）
- 竞品动态（提到的其他公司数据是否有更新？）
- 监管进展（PDUFA 过了吗？FDA 开会了吗？）

标注查询日期。如报告数据与当前事实有出入，在转述中明确标注。

**Step 3 — 识别报告类型，确定转述结构**

| 报告类型 | 识别特征 | 转述重心 |
|---------|---------|---------|
| 卖方深度 initiation/update | 有评级/PT、财务预测、DCF | 核心 thesis、证据链、模型参数、催化剂 |
| 卖方行业/主题报告 | 多公司对比、赛道分析 | 行业逻辑、竞争格局、标的排序 |
| 临床数据解读 | 试验设计、HR/p 值、亚组 | 数据质量、临床意义、跨试验对比 |
| 公司 presentation | 管线表、里程碑、财务 | 公司战略、管线优先级、催化剂日历 |

**Step 4 — 输出结构化转述文档**

固定板块结构（顺序可微调，但核心板块不能缺失）：

```
板块 1：报告元信息
  - 标题、机构、作者、日期、评级/PT
  - 报告时股价 vs 当前股价（联网核实值并列）

板块 2：一句话总结
  - 2-3 句中文概括核心 thesis——领导 10 秒理解主旨
  - 可加一句"领导判断对照"（如果领导对同一话题有过表态）

板块 3-N：按报告逻辑递进的核心内容
  - 行业/疾病背景（为什么这个赛道重要？市场多大？）
  - 科学/技术底层（机制区分、为什么是这个靶点而非那个）
  - 临床证据链（从早期到关键试验的完整数据，表格化呈现）
  - 关键试验设计（人群、统计假设、入组策略、读出时间）
  - 竞争格局（竞品对比表、交易动态、公司在赛道中的位置）
  - 财务/估值（如有）

板块 N+1：报告定位与局限
  - 这是系列中的哪一篇？它覆盖了什么、没覆盖什么？
  - 下一篇会有什么？

板块 最末：关键数据卡片（速查表）
  - 一页看清：股价、市值、评级、催化剂时间、核心假设参数
```

### D.2 转述原则（核心质量规范）

**① 不是翻译，是信息重建**
- 不逐句翻译。原文段落顺序可按中文认知逻辑重组
- 原文 10 页 → 转述 3-5 页（去掉合规套话、disclosures、analyst certification）

**② 补充"暗默知"**
- 报告默认读者已知的背景（如"IL-6 cis vs trans 信号"），用 1-2 句解释
- 报告引用的历史试验（如 CANTOS），简要回顾，让领导不用跳出去查

**③ 表格化对比**
- 原文散落在不同段落的对比数据 → 整理成中文化对比表
- 一张好表胜过三段文字

**④ 标注报告局限**
- 这份报告没说什么？（下一篇才有的内容？没覆盖的风险？）
- 报告假设是否偏乐观/悲观？（如 HR 0.80 的假设在同类试验中处于什么位置？）

**⑤ 文风规则（与模式 B/C 共享）**
```
☐ 直接、朴实、具体——只陈述事实和数字
☐ 不比喻、不排比、不夸张
☐ 原文关键判断保留英文引语 + 中文语境
☐ 每个数据点标注来源
☐ 英文术语保留原文（如 hsCRP、MACE-3、treatment paradigm）
☐ 首次出现时给简注（如"hsCRP（高敏 C 反应蛋白，炎症标志物）"）
```

## 模式 E · 会议纪要

> **触发场景**：用户有一段访谈或会议的语音转文字逐字稿（通常为 docx 格式，含大量口语填充词和识别误差），需要整理成结构化的中文会议纪要。
> **典型用户问题**："帮我弄一份人工纪要""把逐字稿整理成纪要""做一份 Word 纪要"
> **输入**：逐字稿 docx + 参考纪要 docx（1-4 份，用于格式校准）+ 可选 AI 纪要 docx（用于对照查漏）
> **输出**：中文逐字稿 Markdown + Word 人工纪要（楷体 12pt）+ AI 纪要对标分析

### E.0 核心原则：Q 的两种用法

模式 E 中，**Q 不是"问题"，Q 是"这段在说什么"的标签。** 这一点在非 1v1 会议、某一方有大量叙述内容时至关重要。

Q 有两种用法，不可混淆：

| Q 类型 | 来源 | 示例 | 何时用 |
|--------|------|------|--------|
| **真问题** | 逐字稿里有人真的问了 | "XBI产品现在还有额度吗？""三家怎么排序？" | 有人真的提问了 |
| **话题标签** | 逐字稿里没人问，是主讲人在铺陈 | "诺华相关情况""默沙东相关情况""诺华BTK相关" | 主讲人单方面叙述 |

第一种是 Q&A 纪要的自然形态。第二种是轻量 section header——它借用 Q 格式只是为了视觉一致性，不是假装有人问了这个问题。

**典型错误**（2026-07-06 建投医药余老师纪要第一版）：把话题标签写成具体问题。

| ❌ 错误（编造的问题） | ✅ 正确（话题标签） |
|---|---|
| "诺华能否度过专利悬崖危机？相关管线布局和增量空间如何？" | "诺华相关情况" |
| "诺华已上市成长期产品有哪些？增量多大？" | 不单独开 Q，属于"诺华相关情况"下面的 A 段落 |
| "诺华BTK在CSU上为什么获批了还没放量？" | "诺华BTK相关"（然后把放量原因、MS、食物过敏都铺在下面） |

**判断标准**：如果逐字稿里没有人说出这个问句，就不要把它写成 Q。用话题标签代替。

### E.1 多发言人对话追踪：Re Q / Re A / 具名标签

这是模式 E 在处理非 1v1 会议时最重要的机制。当会议有 3+ 人参与、主讲人在铺陈过程中被多次打断/追问/补充时，不能把多个交换合并成一个大 A，也不能给每个追问都起独立的 Q 标题。

**使用三个标签追踪对话链**：

| 标签 | 含义 | 何时用 |
|------|------|--------|
| `Re Q：` | 追问/插话（提问方在同一话题下的跟进） | 提问方在同一个话题内继续追问 |
| `Re A：` | 回应（主讲人对追问的回答） | 主讲人回应追问 |
| `王老师：`（具名） | 第三方专家插入 | 非主讲人、非提问方的第三方发言 |

**示例**（建投医药余老师纪要，IL-6 话题）：

```
Q：诺华IL-6心血管品种相关情况
A：IL-6很重要，但是目前只有二期数据……[余老师介绍]
A：[多段叙述继续]

王老师：安全性之前是对这个产品特别大的风险点……[第三方插入]

Re A：[余老师回应王老师的安全性质疑]

Re Q：[提问方追问 BioAge 投资逻辑]

Re A：[余老师说他不太知道 BioAge]

Re Q：[提问方补充 BioAge 市值、催化剂、与诺和诺德的关系]

王老师：[再次插入，讨论诺华心血管方向收缩]
```

**关键规则**：
- Re Q / Re A 保持对话节奏，但不给每个追问起独立 Q 标题
- 具名标签（王老师：）让读者知道换了发言人
- 所有内容严格按逐字稿时间顺序，不按主题重排
- 同一个话题下的连续追问（如 Lp(a) 线性的 6 轮对话）用 Re 链保留完整推理过程

### E.2 结构模板

```
[日期] [对象]会议纪要              ← 标题，加粗，居中

Key Takeaway                      ← 加粗，无冒号
[主题标签]                        ← 加粗，List Paragraph
  · 要点（List Bullet，一句话一个事实）

── 以下按逐字稿时间顺序 ──

# 叙述块 → 用话题标签
Q：XX相关情况
A：[逐字稿原话，去填充词，不概括]
  [多段 A 继续]
  王老师：[第三方插入]
  Re A：[主讲人回应]
  Re Q：[提问方追问]

# 真问题 → 正常 Q&A
Q：[真实的提问]
A：[真实的回答]
  Re Q：[追问]
  Re A：[回答]
```

### E.3 三个"不要"

1. **不要把话题标签写成具体问题。** "诺华相关情况"可以，"诺华能否度过专利悬崖危机？管线增量如何？"不行——后者假装有人问了这个问题。
2. **不要把多人对话合并成一个大 A。** Re Q / Re A / 具名标签保留对话节奏。读者需要知道信息是一口气交代的还是被追问挤出来的。
3. **不要把叙述内容概括成书面语。** 逐字稿说"辉瑞BD也挺粗糙的"，纪要就写"辉瑞BD粗糙"，不写成"BD决策有效性不足"。逐字稿删掉填充词和口头禅 = 纪要 A 文字。中间没有"概括"这个步骤。

### E.4 判断流程（每次写纪要前过一遍）

```
1. 这是 1v1 访谈还是多人会议？
   → 1v1：传统 Q&A 格式即可
   → 多人/有大量叙述：启用 §E.1 的 Re Q / Re A / 具名标签机制

2. 逐字稿里这句话有人真的问了吗？
   → 有人问 → 真 Q&A
   → 没人问，是主讲人在铺 → 话题标签 "XX相关情况"

3. 这段是主讲人一口气说的还是被追问出来的？
   → 一口气 → 一个话题标签下多段 A
   → 被追问 → Re Q / Re A 保留对话链

4. 有第三方（非主讲人、非提问方）发言吗？
   → 有 → 用 "姓名：" 具名标签标注
```

### E.5 典型案例：建投医药余老师（2026-07-06）

**会议特征**：3 人参与（余老师主讲 + 提问方 + 王老师），余老师边翻 PPT 边介绍诺华/艾伯维/默沙东/AZ，约 70% 内容是单方面叙述，30% 是真实的 Q&A + 多人讨论。

**第一版错误**：
- 把叙述段落全部编成具体问题（"诺华能否度过专利悬崖危机？"等）
- 打乱逐字稿顺序，按自己的逻辑重排
- 把口语概括成书面语

**修正版做法**：
- 叙述块用"诺华相关情况""艾伯维相关情况"等轻量标签
- Lp(a) 线性争议保留 6 轮 Re Q / Re A 对话链
- IL-6 + Factor XI 讨论中王老师的插入用具名标签
- 所有内容严格按逐字稿时间顺序

### E.6 提交前核对清单（模式 E）

```
□ 格式校准完成（参考纪要的字体/字号/加粗/标题格式已提取）
□ 逐字稿全文翻译完成（如为英文），每段标注讲话人和标的公司
□ 联网核实完成（股价/市值/催化剂状态，标日期）
□ Q 的两种用法正确：真问题 vs 话题标签，无混淆
□ 多人对话用 Re Q / Re A / 具名标签追踪，无合并
□ 顺序严格按逐字稿，无重排
□ A 文字 = 逐字稿去填充词和口头禅，无概括改写
□ 箭头(→)计数 = 0
□ 破折号(——)计数 = 0
□ Key Takeaway 无冒号
□ 标题格式：日期 + 一个空格 + 对象 + 会议纪要
□ 字体：楷体 12pt，显式设置含东亚字体回退
□ 生成脚本在验证通过后删除
```
